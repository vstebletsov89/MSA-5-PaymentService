# Task6 - Fitness-функции для OrchestrPay

## Таблица fitness-функций

| Название функции | Назначение | Что проверяется                                                                                                                                       | Действие при нарушении | Способ реализации |
|------------------|------------|-------------------------------------------------------------------------------------------------------------------------------------------------------|------------------------|-------------------|
| SagaStateConsistency | Консистентность состояния Saga | Статус процесса в BPMN-движке соответствует записи в PostgreSQL для каждого `paymentId`                                                               | Создание инцидента, запуск recovery-процесса или перевод в ручной разбор | Scheduled job: сверка состояния процесса и таблицы `payment_saga_state` |
| StuckProcessDetector | Выявление «зависших» процессов | Процессы не находятся слишком долго в промежуточных состояниях (`MANUAL_REVIEW`, `TRANSFER`, `REFUND`) и состояниях ожидания ответа внешних сервисов  | Alert + автоматический перевод в recovery flow | Scheduled job по PostgreSQL + анализ состояния процесса в BPMN |
| ManualReviewDeadlineControl | Контроль SLA и cut-off ручной проверки | Время нахождения в состоянии `MANUAL_REVIEW` не превышает 20 минут; при превышении применяется правило default allow                                  | Автоматическое продолжение процесса (approve) + audit log + alert при аномалиях | Проверка timestamp + BPMN timer event для `Task_ManualReview` |
| PaymentMoneyFlowInvariant | Гарантия целостности движения денег | Инвариант: деньги либо переведены контрагенту, либо возвращены клиенту (нет «подвешенных» состояний)                                                  | Критический alert + запуск reconciliation + автоматический refund при необходимости | Periodic job: сверка состояний reserve / transfer / refund |
| CompensationIntegrity | Контроль корректности компенсаций | При ошибке после pivot-шагов выполнены все компенсирующие действия (`reverse`, `refund`)                                                              | Повторный запуск компенсаций + alert | Проверка истории шагов Saga и статусов операций |
| TimeoutSlaMonitor | Контроль общего SLA платежа | Время выполнения платежей не превышает заданные thresholds (например, 5 секунд для 90%)                                                               | Alert + триггер анализа деградации системы | Метрики Prometheus + alert rules |
| DuplicateProcessingGuard | Обеспечение идемпотентности | Один `paymentId` не обрабатывается более одного раза                                                                                                  | Блокировка дубля + alert | Уникальные ограничения в PostgreSQL + аудит операций |
